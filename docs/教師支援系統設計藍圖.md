# 教師支援系統・設計執行藍圖

> 目標：教師課堂所需一站滿足。前台＝班網（粉蠟筆可愛風，國小生／家長友善），
> 後台＝Notion 資料庫群＋Claude skills 引擎。
> 本文件是後續建造的總參考，2026-07-06 版。

## 一、總體架構

```
                    ┌─────────── 前台：班網（GitHub Pages，粉蠟筆風）───────────┐
                    │ 首頁/聯絡簿/公告/行事曆/週報/相簿/日課表/學習報告/回顧/…   │
                    │ （新增）小小銀行頁・教學駕駛艙入口                         │
                    └──────────────────────△────────────────────────────────┘
                              GitHub Actions 同步（每日3次＋一鍵）
┌── 後台：Notion「🏫 班級經營中心」──────────────────────────────────────────┐
│ 既有：聯絡簿/公告/週報/常用網站/相簿索引/日課表/網站設定/學生學習報告        │
│ 新增：👥 學生名冊 ・ 📝 班經與學習紀錄庫 ・ 🏦 班級銀行帳本 ・ 🏪 班級商店   │
└──────────────────────△────────────────────────────────────────────────┘
                （Claude skills 引擎：讀寫 Notion、生成內容）
  教材生成：NBLMteach / lesson-cockpit / html-slide-builder / teaching-cockpit
  評量：exam-generator（六合一試卷） ・ 作文：essay-grading-batch
  文案：/class-weekly ・ 報告：/class-report（將升級為讀紀錄庫自動生成）
  新增：/class-log（快速記錄） ・ /class-bank（銀行記帳） ・ /lesson-flow（課堂五段流程）
```

核心理念：**老師平時只做一件事——「記錄」**（一句話記下課堂與生活表現），
其餘（分析、評分、建議、報告、班級貨幣）都由系統從紀錄庫自動長出來。

1. **資料中樞缺位**：學習報告目前靠手動填欄位 → 需要「班經與學習紀錄庫」讓報告自動生成。

3. **生活分析未建**：待人處事紀錄 → SEL／輔導理論評分建議，尚無資料結構與生成流程。
4. **教學五段流程是散裝的**：起始評估→教學→差異化→評量→報告，skills 都在但沒有串成一鍵流程、成果沒有回流紀錄庫。
5. **一鍵更新在首頁**：GitHub token 不能放公開網頁，需要免費代理層（見 Phase 5）。
6. **前台風格**：已是糖果色系，可再強化粉蠟筆質感（蠟筆邊框、手繪 icon、圓角加大）。

## 五、設計原則（踩過的坑）

- Notion 免費方案不能用 SQL 查詢（query-data-sources 會 400）→ skills 一律 search＋fetch；
  search 會命中屬性值（如週次），可用來撈整週資料，單次上限 25 筆要補查。
- 靜態站個資一律加密（PBKDF2＋AES-GCM，座號+查詢碼派生金鑰），頭貼 base64 內嵌加密檔。
- Notion 檔案 URL 一小時過期 → 同步時落地或內嵌。
- CI 會自動 commit 資料 → 本機 push 前先 `git pull --rebase`。
- 任何秘密（GitHub token、Notion token）不進前端、不進對話紀錄，只進 GitHub Secrets／
  老師自己的捷徑／Apps Script。
- **模型無關性**：系統建好後，換用較小模型（如 Sonnet）跑 skills 也要有同等品質。
  作法：把判斷邏輯寫死在 skill 文件裡——評分用明確公式（如報告五向度「基準 3 ± 正負×程度×0.5」）、
  解析用明確規則表（類別／正負向的關鍵詞對應）、輸出用固定欄位範本與硬性禁令（禁複製上週文字），
  讓模型只做「套規則」而非「臨場發揮」；新 skill 驗收時要問「這條指令 Sonnet 會不會理解歧義」。
